Chapter 01

Game Vision & Overview

What THRESHOLD is, who it's for, and why it exists.

Concept Statement

THRESHOLD is a top-down 3D mobile shooter/roguelite for Android where no two runs are ever the same — and neither are the enemies. Powered by a 5-agent Gemini AI system that observes every player decision, adapts the corridors they move through, retunes enemy tactics in real time, and even allows enemies to defect to the player's side when they recognise they can't win.

One-line pitch: A roguelite where the AI doesn't just generate content — it understands the player and reshapes every run around their skill, strategy, and emotional state.

Design Pillars

Adaptive

The game reshapes itself around the player's skill, pacing, and strategy — not a difficulty slider.

Reactive AI

Enemies observe and counter the player's dominant tactics in real time via Gemini-driven reasoning.

Procedural Depth

Every corridor, encounter, and enemy mix is generated fresh per run and validated for fairness.

Agentic Clarity

Every AI decision comes with a human-readable reasoning trace — fully inspectable, fully auditable.

Chapter 02

Core Game Loop

The six-step cycle that drives every run from cold open to post-run debrief.

1

Start Run

Director Agent reads last-run metrics and generates a difficulty profile. Level Gen Agent produces a corridor configuration. QC Agent validates the layout for solvability. Run begins.

2

Enter Corridor

Player enters procedurally stitched rooms. Enemies spawn according to the density setting from the Director. PlayerMetricsTracker begins recording all actions.

3

Encounter

NPC Brain Agent evaluates the player's strategy every 20 seconds and updates enemy tactics — choosing from Flank, Suppress, Rush, Retreat, or Defect. Random events may trigger mid-encounter.

4

Room Clear

Player clears all enemies. Accuracy, time-to-clear, health remaining, and resource usage are logged. Difficulty is adjusted at the micro-level for the next room.

5

Threshold Reached

The run ends when a termination condition is met. Reward Agent evaluates the player's effort and produces XP, bonuses, and incentive messaging.

6

Between Runs

Director reads full session data and produces a next-run profile with reasoning trace. The player sees a summary screen with an agent-authored explanation of what changed and why.

Run Termination Conditions

ConditionTriggerAgent Involved
Kill QuotaAgent-set target reachedDirector
Floor Count5–12 rooms completedDirector
Player DeathHealth reaches zero—
Session FatigueDetected via engagement signalsDirector
Boss EliminationFloor boss defeatedDirector / NPC Brain
Chapter 03

Gemini Agent Architecture

Five specialised Gemini agents collaborate to drive every aspect of the player experience.

Communication Layer

All five agents call the Gemini API at runtime via GeminiAgentBridge.cs (async UnityWebRequest). Each receives a JSON game-state payload and returns a decision JSON with a mandatory 5-step reasoning trace.

A1
Director Agent
Orchestrator — Gemini Pro

Observes: Deaths, kills, accuracy %, time-per-room, retry count, session length, quit events

Decides: Difficulty multiplier (0.5×–2.5×), room count, enemy archetypes, event probability

A2
Level Gen Agent
Generator — Gemini Pro

Observes: Difficulty profile, player strategy patterns, last-run layout

Decides: Room sequence, branching paths, spawn zones, item placement, event triggers

A3
NPC Brain Agent
Behaviour — Gemini Flash

Observes: Player position, weapon, movement, kill streak, health, dominant tactic

Decides: NPC state (PATROL / ATTACK / FLANK / SUPPRESS / RETREAT / ALLIED) — re-evaluated every 20 seconds

A4
QC Agent
Validator — Gemini Pro

Observes: Generated level config — room graph, spawn points, solvability constraints

Decides: ACCEPT or REJECT with reason; failure note returned to Level Gen for regeneration

A5
Reward Agent
Evaluator — Gemini Pro

Observes: Run outcome, effort signals, improvement delta, retention risk, behavioural anomalies

Decides: XP amount, bonus multiplier, unlock suggestion, incentive message

Also performs: Fair play and anti-exploit checks — AFK detection, suspicious accuracy patterns, and input anomaly flagging. Can suggest optional next-run challenges (e.g. "Clear 3 rooms without healing for 2× XP bonus").

Chapter 05

Adaptive Difficulty System

How the Director Agent reasons over 7 player signals to tune every run — no difficulty slider, no static rules.

Player Signal Matrix

The Director Agent ingests 7 real-time player signals (hackathon requirement: 5+). Each signal has a weight priority and a behavioural response.

SignalPriorityInterpretation & Response
Completion TimeHighRoom timer vs par. Fast = increase density. Slow = reduce spawns.
Accuracy %HighAbove 85% sustained = add evasive enemy archetypes.
Death CountHigh3+ deaths = decrease difficulty + inject contextual hints.
Retry BehaviourMediumQuick retries = frustration detected. Widen corridors, reduce ambushes.
Resource EfficiencyMediumLow ammo per kill = skilled player. Trigger elite archetype spawns.
Session LengthLowShort sessions + rage-quit pattern = reduce difficulty on next cold open.
Win/Loss StreakHigh3 wins = guaranteed spike. 3 losses = guaranteed ease-off.

Agentic vs. Rule-Based — The Critical Distinction

This is AGENTIC, not rule-based. A statement like if deaths > 3: difficulty -= 1 is not acceptable. The agent reasons over all 7 signals holistically to produce a nuanced decision.
Example: Agentic Reasoning in Action

"Player died 4 times but accuracy was 89% and retry was immediate — the layout was likely unfair, not the player. Recommend restructuring room topology rather than reducing enemy count. Deaths were concentrated in Room 4 (choke point with no flanking option). Widening the corridor by one tile and adding a second cover position should resolve without reducing challenge."

This contrasts with a naive rule that would simply reduce enemy count — missing the root cause entirely. The agentic approach considers why the player struggled, not just that they struggled.

Mid-Run Micro-Adjustment

Between rooms, a lightweight local system (no Gemini call) can apply small tweaks to the next room's spawns. This complements the full Director Agent reasoning that happens between runs.

Examples of Micro-Adjustments
  • Player health below 30% → inject a health kit into the next room's loot table
  • Clear time significantly under par → add +1 enemy to the next room's spawn count
  • Ammo critically low → place an ammo cache near the room entrance
Note: Micro-adjustments are deterministic, threshold-based, and deliberately limited in scope. The Director Agent handles all holistic, multi-signal reasoning at the run boundary.
Chapter 06

NPC Behaviour & Allegiance System

A 6-state machine driven by the NPC Brain Agent — not hard-coded transitions, but reasoned AI decisions.

NPC State Machine

PATROL
Default. Follows waypoints.
ATTACK
Direct fire when player is exposed.
FLANK
Routes to off-axis position when player is stationary.
SUPPRESS
Rapid fire from cover when player is reloading / low health.
RETREAT
Falls back when NPC health is critical.
ALLIED ★
Defection — NPC joins the player's team.

Defection: The Signature Feature

When the NPC Brain Agent determines an NPC's survival probability is near zero AND the player hasn't exploited downed enemies, it can trigger defection. The NPC lowers its weapon, a UI prompt appears, and the NPC begins attacking its former allies.

Defection is probabilistic — the Brain Agent also weighs enemy team state, current difficulty tier, and whether adding an ally would make the encounter trivially easy.

Defection Trace Example

NPC Brain Agent — 5-Step Defection Trace
StepContent
ObserveKill streak 5+. NPC-04 health < 20%. Player hasn't exploited downed enemies. 3 allied NPCs remain.
InferContinued opposition = certain death for NPC-04. Defection = survival. Player behaviour suggests mercy-oriented playstyle.
DecideTrigger defection for NPC-04. Weighed against: ally count (0 current allies), difficulty tier (1.3×), team morale impact.
ActState → ALLIED. Weapon lowered animation. UI prompt: "NPC-04 has surrendered and joined your side."
EvaluateDid defected NPC help player survive next 2 rooms? Was encounter difficulty still meaningful?

NPC Archetypes

TypeHPFire RateAccuracySpeedRole
GRUNT1001.0 shots/s40%NormalBaseline infantry, cannon fodder in large groups
FLANKER801.2 shots/s50%FastRepositions aggressively, exploits player blind spots
SUPPRESSOR1202.0 shots/s25%SlowHigh-volume fire from cover, pins player in position
ELITE2501.5 shots/s60%NormalBoss-level threat, deployed sparingly by Director Agent

Defection Constraints

Hard Limits on Defection
  • Maximum 1 defection per room — prevents mass surrender
  • Maximum 2 defected allies per run — preserves challenge integrity
  • ELITE NPCs almost never defect unless they are the last enemy standing
  • No defection if it would make the encounter trivially easy (Brain Agent evaluates remaining enemy strength)
Chapter 07

Procedural Level Generation

Every floor is unique — generated by AI, validated by AI, and instantiated by Unity.

Generation Pipeline

  1. Director Agent produces a difficulty profile (multiplier, room count, archetype mix).
  2. Level Gen Agent creates a room graph — sequence, branching, spawn zones, items, events.
  3. QC Agent validates the layout. Returns ACCEPT or REJECT with reason. Maximum 3 rejection attempts; then fallback to a pre-validated safe layout.
  4. FloorGenerator.cs instantiates rooms in Unity from the validated configuration.

Room Prefab Shapes (5 Physical Types)

ShapeDoorwaysLayout Description
STRAIGHT2 (opposite)Linear corridor — simplest traversal path
T-JUNCTION3Branching path — creates route decisions
CROSSROADS4Open intersection — maximum connectivity
CORNER2 (adjacent)L-shaped bend — restricted sightlines
DEAD END1Terminal branch — forces backtracking

Shape vs. Role Separation

A room's physical shape (which prefab) is independent from its gameplay role (what happens inside). The same T-Junction can be a calm pacing room or a deadly ambush — the Level Gen Agent decides the role assignment based on the Director's difficulty profile and the player's recent performance.

Gameplay Roles (8 Types)

RoleDescription
ENTRYPlayer spawn point. Safe, no enemies. Establishes orientation.
EXITRun endpoint, reached after boss encounter. Triggers run summary.
PACINGBreathing room between fights. Low threat, resource recovery.
COMBATStandard enemy encounter. Core gameplay loop.
AMBUSHSurprise spawns triggered by player movement or position.
BOSSElite encounter. Hardest room in the run, typically ELITE NPC.
LOOTOptional branch with rewards or traps. Risk/reward decision.
CHOKENarrow tactical room favouring suppression-style combat.
Example: A CORNER prefab assigned the AMBUSH role spawns enemies behind the L-bend when the player rounds the corner. The same CORNER prefab assigned the LOOT role places a reward cache around the bend with a trap trigger.

Chapter 09

Baseline Comparison

FixedRuleDirector mode provides the +5% hackathon bonus by demonstrating measurable agentic improvement.

Fixed Rule Baseline

FixedRuleDirector — Static Rules (No Context)
  • if deaths > 3: difficulty -= 1 — no consideration of why the player died
  • if run_won: difficulty += 1 — ignores how easily the player won
  • NPCs: ATTACK state always — no tactical adaptation
  • Levels: random selection from 5 fixed templates
  • Rewards: flat XP per kill — no effort evaluation

Chapter 11

Development Roadmap

Five sequential phases from workspace setup to final submission. Skills files updated between phases so later agents inherit architecture.

Phase 1 — Foundation

Install Antigravity, configure Unity workspace, define Skills files, build GeminiAgentBridge.cs, verify API connectivity end-to-end.

Phase 2 — Core Systems

Spawn parallel build agents for ProceduralRoomGenerator, PlayerMetricsTracker, NPCStateMachine. Wire all systems to the bridge. Director Agent end-to-end. First playable run achieved.

Phase 3 — Agent Intelligence

Level Gen + QC Agent integrated with reject/regen loop. NPC Brain Agent with all 6 states + defection. Reward Agent operational. Debug panel with live traces.

Phase 4 — Baseline & Polish

FixedRuleDirector baseline implemented. Comparison metrics captured. UI polish (camera, joystick, HUD). Mobile performance pass (ASTC textures, draw call batching, 60fps target).

Phase 5 — Delivery

Demo video, Antigravity workflow video, trace export, README, architecture map, APK build, device testing, final submission.

Chapter 12

Privacy & Cost Notes

Data handling, API cost projections, and latency targets.

Data Privacy

  • No raw personal data is stored. All player signals are numeric, session-scoped, and anonymised before any Gemini API call.
  • Development and testing use mock data exclusively.
  • No data leaves the device except anonymised JSON payloads to the Gemini API.
